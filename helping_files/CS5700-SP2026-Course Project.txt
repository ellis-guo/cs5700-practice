Secure Reliable File Transfer (SRFT)
Your task is to write a client program called SRFT_UDPClient that takes a filename as input. Then, the client program sends the filename to a UDP web server to request its download. The web server program is called SRFT_UDPServer. Your client and server programs must use SOCK_RAW as the type of their sockets, that is, you are responsible for building the IP and UDP headers in each packet.


As you know, UDP does not provide reliable data transfer and you cannot guarantee that the received file is the same as the sent file. This means that the file may have been modified in transit (e.g., some bits may be corrupted, some chunks may be lost, etc.). Therefore, the goal of this project is to provide secure and reliable file transfer for a file of any size and any type using UDP.

Phase2: Secure and Reliable File Transfer
In addition to reliability, your protocol must provide secure file transfer. Your implemented protocol in Phase1 does not provide confidentiality, integrity (limited to checksum), or authentication, so an attacker on the network could (1) read file contents, (2) modify packets in transit, (3) inject forged packets, or (4) replay old packets. Your task is to add a security layer above UDP (inside the UDP payload) while still using SOCK_RAW and constructing IP and UDP headers manually.
Your secure extension must provide the following properties:
1. Confidentiality: File contents must be encrypted in transit.

2. Integrity: Any tampering with packets must be detected.

3. Authentication: Client and server must authenticate each other to prevent packet injection by third parties.

4. Replay protection: Old packets must not be accepted if replayed.

Key Management (Pre-Shared Key)
You will use a pre-shared key (PSK) known to both the client and server. The PSK may be stored in a configuration file.
Example: PSK = b"replace-with-32+bytes-random-key"
Security Handshake (Session Establishment)
Before the file transfer begins, the client and server must perform a short handshake to establish a fresh session and derive per-session keys from the PSK.
   1. ClientHello (Client → Server)

      * Client generates a random client_nonce (e.g., 16 bytes).
      * Client sends client_nonce and protocol version/cipher info.
      * Client includes an HMAC (Hash-based Message Authentication Code) computed using the PSK over the handshake fields.

         2. ServerHello (Server → Client)

            * Server verifies the client HMAC.
            * Server generates a random server_nonce (e.g., 16 bytes) and a random session_id (e.g., 8 bytes).
            * Server replies with server_nonce, session_id, and an HMAC using the PSK.
After the handshake, both sides derive session keys using a Key Derivation Function (KDF), e.g., HKDF-SHA256:
            * enc_key It is used to encrypt and authenticate data packets using Authenticated Encryption with Associated Data (AEAD).

            * optionally ack_key (used to authenticate ACK-only packets if you choose not to encrypt ACKs)

If any handshake HMAC verification fails, the connection must be rejected.
Secure Packet Protection (AEAD Encryption)
All file DATA packets must be protected using an authenticated-encryption scheme such as AES-GCM (AEAD):
               * Each DATA packet uses a unique nonce/IV (e.g., 12 bytes for AES-GCM).

               * The encryption must authenticate the packet metadata using AAD (Additional Authenticated Data) so attackers cannot alter control fields without detection.

At minimum, the following fields must be authenticated (as AAD):
                  * Session_id
                  * Sequence_number
                  * ack_number (if present)
                  * flags/type
If decryption/authentication fails for a received packet, the receiver must treat it as invalid and discard it (it must not be delivered to the file output).
You should encrypt/authenticate ACK packets with the same AEAD mechanism as DATA packets.

Replay Protection
Your receiver must implement replay protection using sequence numbers or authenticated packets:
                  * Reject packets with sequence numbers that have already been accepted and written (or already buffered as received).

                  * Packets outside the receiver window may be discarded.
End-to-End File Verification (Required)
To ensure the entire file was correctly received, the server and client must verify a final file digest:
                     * The sender computes SHA-256(file_bytes) and sends it in a final control packet protected by AEAD.

                     * The receiver computes SHA-256 on the reconstructed file and compares digests.

                     * The transfer is “successful” only if the digests match.

Your updated server program for Phase2 should produce a text output file and reports the results using the following format:

                        * Security enabled (PSK + AEAD): Yes/No
                        * Handshake status: Success/Fail
                        * AEAD authentication failures (invalid packets dropped): <count>
                        * Replay drops (duplicate/out-of-window packets): <count>
                        * SHA-256 match: Yes/No
